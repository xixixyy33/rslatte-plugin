# 同步与归档

## 概述

RSLatte 里与「同步」相关的概念分两层：

1. **本地索引**：在 Vault 内（中央索引目录、按空间分桶）维护 `*-index.json` 等文件，由 **Pipeline 引擎** 的增量扫描、全量重建、归档与（可选）**reconcile** 驱动。  
2. **后端数据库同步（可选）**：在配置合法 **`apiBaseUrl`**、**`vaultId`** 且各模块打开 **`enableDbSync`** 时，将部分数据 upsert 到服务端；请求头携带 **`X-Space-Id`**（与当前空间一致，切换空间时 **`api.setSpaceId`** 会更新）。

「归档」则包括：**索引分片/迁出**（闭环条目按阈值迁入 `archive/` 等）、以及 **笔记文件**层面的移动（如输出、联系人等模块按钮说明中的「笔记归档」），具体行为因模块而异。

实现主线：**`pipelineEngine.runE2(spaceCtx, moduleKey, mode)`**（`src/services/pipeline/pipelineEngine.ts`）、各模块 **Atomic Spec**（`src/services/pipeline/specs/*`）、打卡/财务等 **`recordSync`**（`src/plugin/recordSync.ts`）。

## Pipeline 模式（`RSLattePipelineMode`）

与 **`gates.evaluateReconcileGate`**、**`runE2`** 共用一套命名（见 `src/services/pipeline/types.ts`）：

| 模式 | 典型触发 | 摘要 |
|------|----------|------|
| **`auto` / `auto_refresh`** | 定时协调器 tick | 增量更新索引；**不**走与手动刷新相同的 reconcile 策略（自动轨与手动轨分离，见协调器注释） |
| **`auto_archive`** | 定时 | 按模块策略做自动归档相关步骤 |
| **`manual_refresh`** | 侧栏 **🔄**、部分命令 | 增量扫描 + 写索引；若模块开启 DB 同步且门控通过，会 **buildOps → flushQueue**，再按 **`RSLatteReconcileGate`** 决定是否 **`reconcile`**；通过时还可执行 **`archiveOutOfRange`**（引擎内 P6 规则） |
| **`manual_archive`** | 侧栏 **🗄** 等 | 归档相位；部分模块与 **record 手动互斥**（避免与打卡/财务忙时冲突） |
| **`rebuild`** | 侧栏 **🧱** | 全量重建索引；门控通过时可 reconcile / 归档 |

**锁粒度**：`runE2` 推荐使用 **`SpaceCtx`**（含 **`vaultId` + `spaceId`**），互斥键包含空间，避免 **多空间并行跑同一模块** 时串扰。未传 `ctx` 时引擎会告警（锁可能不含 space）。

## Reconcile 门控（`RSLatteReconcileGate`）

定义见 **`src/services/pipeline/moduleSpec.ts`**。引擎在 **manual_refresh / rebuild** 等路径上根据 **`getReconcileGate`** 返回值决定是否调用 **`spec.reconcile`**。字段含义包括：

- **`dbSyncEnabled`**：本模块是否启用 DB 同步（未启用则通常跳过与队列相关的 reconcile）。  
- **`allowReconcileWithoutDbSync`**：如健康等模块可 **仅做本地索引校准** 仍允许 reconcile。  
- **`pendingCount` / `failedCount`**：同步队列是否排空。  
- **`deltaSize`**：变更量过大时可能建议走 rebuild。  
- **`uidMissingCount` / `parseErrorCount` / `dirtyCount`**：索引脏数据检测。

各模块对 gate 的计算方式不同（任务/提醒可参考索引内 **`dbSyncState`** 统计等）。

## 后端同步（按模块）

**全局前提**（常见模式）：**合法 `http(s)://` 的 `apiBaseUrl`**、**`vaultId`**；后端可达性由 **`VaultService`** 的 **`ensureVaultReadySafe` / `checkDbReadySafe`**、`shouldTouchBackendNow` 等统一门控（避免无效请求刷屏）。

**空间**：当前空间 ID 与库内记录 **`space_id`** 对齐；切换空间见 [空间与索引](空间与索引.md)。

**模块级开关**（需在设置中分别打开，且模块本身已启用），包括但不限于：

- **打卡 / 财务**：`checkinPanel.enableDbSync`、`financePanel.enableDbSync`；插件对外 **`isRSLatteDbSyncEnabled()`** 定义为 **二者其一开启** 即视为 record 方向「DB 同步可用」的粗粒度判断。  
- **健康**：`healthPanel.enableDbSync`（`health_records`）。  
- **任务 / 提醒 / 输出 / 项目 / 联系人**：各自 Panel 或模块设置中的 **`enableDbSync`**（与侧栏 **DB 状态灯**、**`_dbSyncMeta`** 一致）。  
- **知识库**：`knowledgePanel.enableDbSync`（`knowledge-index` → 后端表）。  
- **WorkEvent**：`workEventDbSyncEnabled` 等与操作日志入库相关项。

**打卡/财务附加能力**（`recordSync`）：例如 **打卡项/财务分类清单** 的 **`autoSyncRecordListsToDb`**（内容变更后、去重 key 未变则跳过）；**手动刷新**路径下 Pipeline 可在满足条件时调用 **`syncRecordIndexToDbNow`** 等。

**UI**：各侧栏标题旁的 **同步状态灯**、tooltip；工作台可监听 **`rslatte:db-sync-status-changed`** 刷新展示（见 `constants/space.ts`）。

## 索引自动刷新

- 设置项：**`autoRefreshIndexEnabled`**、**`autoRefreshIndexIntervalMin`**（分钟，**全局**，不按空间切换；见 [空间与索引](空间与索引.md)）。  
- 调度由 **`main.ts`** 侧定时器驱动，**协调器**对启用模块执行 **`runE2(..., auto_refresh)`**（与侧栏手动 **`manual_refresh`** 同源引擎、不同模式策略）。  
- 大库可适当 **加大间隔**，减少 IO 与 CPU；与 **手动 🔄** 互补。

## 归档

### 自动归档（配置项）

多数与索引/日记相关的模块在设置中有 **`autoArchiveEnabled`** 与 **`archiveThresholdDays`**（默认常见为 **90** 天，具体以 **`defaults.ts` / `clampModuleArchiveThresholdsInSettings`** 与各 Spec 为准）。超过阈值的 **已闭环** 条目会在合适时机 **迁入归档索引分片** 或等价逻辑，避免主索引无限膨胀（**日记正文位置**是否移动取决于模块：有的仅迁索引，有的同时移动笔记，以侧栏按钮 **title** 与 Spec 为准）。

### 手动归档

侧栏 **🗄** 触发 **`runE2(..., manual_archive)`**，通常处理「已超过阈值且已闭环」的批次；**record** 类可能与 **打卡/财务正在执行的手动操作** 互斥（`main.ts` 中 **`tryBeginRecordManualOp`** / **`isRecordManualBusy`** 等）。

### 与「同步」的关系

**`manual_refresh`** 在 reconcile 门控允许时可能附带 **`archiveOutOfRange`**（引擎内规则）；**自动刷新轨**不替代完整的手动归档按钮语义，具体以各模块实现为准。

## 操作互斥与提示

- **输出模块**：**`_outputOpInFlight`** 区分自动/手动，避免并发写。  
- **打卡/财务**：手动刷新/归档与 **全库自动刷新 tick** 互斥时会 **Notice** 提示稍后再试。  
- 失败时除 Notice 外可查 **控制台日志**；DB 失败项可能在 **`output-sync-state.json`**、索引条目的 **`dbSyncState`** 等留有重试信息。
